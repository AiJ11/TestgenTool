────────────────────────────────────────────────────────────────────────────────
  MUTATION OPERATORS — SELECTION RATIONALE
────────────────────────────────────────────────────────────────────────────────

  The six operators were chosen based on three criteria:
  (a) industry-standard coverage as defined by Andrews et al. (2005) and the
      Offutt & Untch (2001) catalogue of first-order mutation operators;
  (b) alignment with the most common defect types observed in REST API backends
      (wrong status codes, inverted guards, missing side-effects); and
  (c) feasibility — operators that produce syntactically valid, compilable
      JavaScript mutants without manual AST manipulation.

  ┌──────┬────────────────────────────────────┬───────────┬────────────────────────────────────────────────────────────┐
  │  Op  │ Full Name                          │  Count    │ Why Chosen for GhostSocket                                 │
  ├──────┼────────────────────────────────────┼───────────┼────────────────────────────────────────────────────────────┤
  │ ROR  │ Relational Operator Replacement    │  8 (15%)  │ GhostSocket has several critical comparisons:              │
  │      │                                    │           │  • Temporal guards (expiry < now) — off-by-one here        │
  │      │                                    │           │    can allow sessions to be joined after expiry.           │
  │      │                                    │           │  • Ownership checks (userId !== caller) — flipping to      │
  │      │                                    │           │    === means the owner is blocked and attackers succeed.   │
  │      │                                    │           │  • Permission length checks (=== 0 vs !== 0).              │
  │      │                                    │           │ ROR is the most studied operator (Hamlet 1977); off-by-one │
  │      │                                    │           │ boundary errors are among the most common real bugs.       │
  ├──────┼────────────────────────────────────┼───────────┼────────────────────────────────────────────────────────────┤
  │ LCR  │ Logical Connector Replacement      │  2  (4%)  │ Guards combining &&/|| are common in Express middleware.   │
  │      │                                    │           │ Replacing || with && weakens composite guards (e.g.,       │
  │      │                                    │           │ !session || terminated becomes !session && terminated,     │
  │      │                                    │           │ causing a null-pointer crash when session is absent).      │
  │      │                                    │           │ Only 2 such operators appear in the source; applying LCR   │
  │      │                                    │           │ to all of them gives complete structural coverage.         │
  ├──────┼────────────────────────────────────┼───────────┼────────────────────────────────────────────────────────────┤
  │ AOR  │ Arithmetic Operator Replacement    │  4  (7%)  │ Two arithmetic expressions control security-critical       │
  │      │                                    │           │ timing: the 10-minute expiry window for sessions and the   │
  │      │                                    │           │ 5-minute OTP lifetime. Replacing + with - makes OTPs/      │
  │      │                                    │           │ sessions immediately invalid (always rejects). Changing    │
  │      │                                    │           │ the constant (10→9, 5→50) weakens or tightens the policy.  │
  │      │                                    │           │ AOR is the second most effective operator in Offutt's      │
  │      │                                    │           │ studies; arithmetic errors are common in date arithmetic.  │
  ├──────┼────────────────────────────────────┼───────────┼────────────────────────────────────────────────────────────┤
  │ CDL  │ Constant / Literal Replacement     │ 22 (41%)  │ Largest operator class because GhostSocket returns         │
  │      │                                    │           │ explicit HTTP status codes (201, 200, 400, 403, 404) and   │
  │      │                                    │           │ uses role strings ("owner", "user") throughout. These      │
  │      │                                    │           │ constants are the primary observable output of each        │
  │      │                                    │           │ endpoint and are exactly what TestGen's postconditions      │
  │      │                                    │           │ assert. CDL directly simulates the most common real        │
  │      │                                    │           │ developer mistake: returning the wrong status code.        │
  │      │                                    │           │ Mresa & Bottaci (1999) show CDL achieves high coupling     │
  │      │                                    │           │ with real fault distributions in API code.                 │
  ├──────┼────────────────────────────────────┼───────────┼────────────────────────────────────────────────────────────┤
  │ SDL  │ Statement Deletion                 │ 15 (28%)  │ Missing side-effects (a DB write, a session mark, a link   │
  │      │                                    │           │ deletion) are the second-most-reported bug class in REST   │
  │      │                                    │           │ APIs (Segura et al. 2018). SDL models "I forgot to write   │
  │      │                                    │           │ this line" exactly. It also models security omissions:     │
  │      │                                    │           │ deleting an auth check removes an entire guard. TestGen's  │
  │      │                                    │           │ stateful multi-step tests are uniquely well-positioned to  │
  │      │                                    │           │ catch SDL mutants because a subsequent API call in the     │
  │      │                                    │           │ sequence will observe the missing state change.            │
  ├──────┼────────────────────────────────────┼───────────┼────────────────────────────────────────────────────────────┤
  │ UOI  │ Unary Operator Insertion           │  3  (6%)  │ Negating a boolean guard (if (!x) → if (x)) is a clean    │
  │      │                                    │           │ single-character change that completely inverts the        │
  │      │                                    │           │ access-control decision. We applied UOI only to the three  │
  │      │                                    │           │ most security-critical null checks (ownership in create-   │
  │      │                                    │           │ Session, terminateSession, getDeviceInfo) rather than all  │
  │      │                                    │           │ null checks, to avoid equivalent mutants (negating a       │
  │      │                                    │           │ non-observable null check produces a trivially equivalent  │
  │      │                                    │           │ mutant with identical test output).                        │
  └──────┴────────────────────────────────────┴───────────┴────────────────────────────────────────────────────────────┘

  Operators NOT used and why:
    SOR (Shift Operator Replacement): no bitwise shifts in JavaScript REST logic.
    COR (Conditional Operator ?:):    no ternary operators in security-critical paths.
    BSR (Bit-Shift Replacement):      not applicable to this codebase.
    SVR (Scalar Variable Replacement): would often produce equivalent mutants in this
                                       code because many variables have identical type/scope.

  STANDARD MUTATION OPERATORS APPLIED
────────────────────────────────────────────────────────────────────────────────

  ROR  Relational Operator Replacement    8 mutants
       Replace <  with >,  <=  with >=,  ==  with !=,  !==  with ===
       Also: off-by-one replacements (< → <=)

  LCR  Logical Connector Replacement      2 mutants
       Replace &&  with ||,  ||  with &&

  AOR  Arithmetic Operator Replacement    4 mutants
       Replace +  with -,  multiply constants (× 10)

  CDL  Constant / Literal Replacement    22 mutants
       Replace HTTP status codes (201→200, 403→200, 404→200, 400→200)
       Replace string literals ("owner"→"user", role filter changes)
       Replace boolean literals (terminated:true → terminated:false)

  SDL  Statement Deletion                15 mutants
       Delete entire validation blocks, guard clauses, or DB operations

  UOI  Unary Operator Insertion           3 mutants
       Negate a boolean condition: if (!x) → if (x)

────────────────────────────────────────────────────────────────────────────────
  ANALYSIS: WHY TESTGEN KILLS ALL 54 MUTANTS
────────────────────────────────────────────────────────────────────────────────

1. SPECIFICATION-DRIVEN POSTCONDITIONS
   TestGen generates assertions directly from the formal API specification.
   Any mutation that changes an HTTP status code (CDL) or inverts a guard (ROR/UOI)
   is caught because the postcondition "_result = 201" (or 200, 403, 404) is
   compared against the backend response in every test.

2. STATEFUL MULTI-STEP TESTS
   Tests like test17_fullSessionLifecycle exercise 6 API calls in sequence.
   Mutations that corrupt state (SDL: skip DB write, CDL: wrong terminated flag)
   are caught because a later step in the sequence observes the corrupted state.

3. NEGATIVE / FORBIDDEN TESTS (test13–16)
   Four dedicated negative tests verify that forbidden actions return 403/404.
   Any mutation that changes those error codes (M02, M15, M21, M29, M38) or
   removes the guard (M22, M30) is immediately killed by these tests.

4. BOUNDARY VALUE COVERAGE
   TestGen creates sessions with exact 10-minute expiry windows. Off-by-one
   mutations (M06, M44) and sign-flip mutations (M07, M52) on temporal
   calculations are caught because the generated test value is exactly at
   the boundary.

5. OPERATOR COVERAGE ACROSS 25 TESTS
   With 25 tests spanning all endpoints (register, device, session, join,
   terminate, update, delete, get-other), every controller path is exercised.
   SDL deletions are caught because follow-up assertions in the same or later
   tests depend on the side effect that was deleted.
