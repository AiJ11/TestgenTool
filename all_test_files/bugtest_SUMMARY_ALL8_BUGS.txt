HOW TESTGEN DETECTS BUGS — MECHANISM
================================================================================

  1. SPEC → ATC (Abstract Test Case)
     Formal spec with preconditions and postconditions compiled into a program
     with assume() and assert() statements.

  2. ATC → RewriteGlobalsVisitor → TEST-API ATC
     Global map ops (U[k]=v) rewritten to get/set test-API calls that interact
     with live backend state.

  3. TEST-API ATC → SEE (Symbolic Execution Engine)
     First iteration: symbolic variables (X0,X1,...) for all inputs.
     Z3 finds concrete values satisfying all assume() constraints.

  4. Concrete values → Backend execution
     Second iteration with Z3-assigned values calls the actual backend.
     assert() evaluated against real responses. ASSERT FAILED = BUG DETECTED.

  5. CTC (Concrete Test Case) generated showing full sequence with HTTP calls.

DETECTION STRATEGY PER BUG TYPE
================================================================================

  Status code bugs (B1, B2, B3):
    → Add _result == ExpectedCode to spec postcondition.
    → Factory returns resp.statusCode; spec fails if code is wrong.

  Arithmetic / calculation bugs (B4, B8):
    → Add a verification helper endpoint to testRoutes.js (check_order_amount,
      check_cart_total). Endpoint does the correct math and returns 1 or 400.
    → New spec block calls the verifier; postcondition asserts _result == 1.

  Missing side-effect bugs (B5):
    → Spec postcondition checks the state that SHOULD have changed.
    → get_C() fetches live backend state; missing delete is caught directly.

  Boundary condition bugs (B6):
    → New spec block sends the boundary value (quantity=0) to the update endpoint.
    → Postcondition asserts _result == 400 (rejection expected).
    → Bug bypasses guard → DB error (500) or success (200), both ≠ 400 → CAUGHT.

  Wrong conditional guard bugs (B7):
    → Test sequence exercises the full state machine to reach the guarded state.
    → Postcondition on the guarded action asserts _result in dom(correctStateMap).
    → Bug causes wrong HTTP response → assertion fails.
