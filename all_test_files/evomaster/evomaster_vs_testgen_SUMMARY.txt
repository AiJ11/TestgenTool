======================================================================

======================================================================
WHY TESTGEN ACHIEVES OMTSP = 1.00
======================================================================

TestGen explicitly models:
  1. Preconditions: "to call addToCart, a product must exist (from createProduct)"
  2. Postconditions: "after createProduct, productId is stored in cache P"
  3. Execution order: test strings execute calls in dependency order
  4. Response extraction: product IDs, tokens, order IDs are extracted from
     HTTP responses and passed into subsequent calls via the sigma map (σ)

EvoMaster's limitations:
  1. No response chaining: each call uses random/hardcoded values
  2. No precondition awareness: doesn't know register must precede login
  3. Cannot handle external OAuth (Clerk): TripVault/GhostSocket = 0%
  4. JWT token problem: even if register/login bodies were valid, EvoMaster
     doesn't extract the JWT from the login response for subsequent calls
  5. Evolutionary search cannot overcome logical API ordering requirements

======================================================================
CONCLUSION
======================================================================

EvoMaster, despite being a state-of-the-art evolutionary REST API tester,
achieves an average OMTSP of only 0.08 across the 6 tested backends,
compared to TestGen's 1.00. The fundamental limitation is that black-box
evolutionary tools do not model API call dependencies — they cannot extract
values from one response and pass them correctly to the next call.

TestGen's specification-based approach using formal preconditions and
postconditions completely solves this problem, enabling 100% test string
execution depth on all 6 backends.
