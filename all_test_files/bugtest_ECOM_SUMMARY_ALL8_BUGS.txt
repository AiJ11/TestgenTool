HOW TESTGEN DETECTS BUGS — MECHANISM
================================================================================

  1. SPEC → ATC (Abstract Test Case)
     Formal spec with preconditions and postconditions compiled into a program
     with assume() and assert() statements.

  2. ATC → RewriteGlobalsVisitor → TEST-API ATC
     Global map ops (U[k]=v) rewritten to get/set test-API calls that interact
     with live backend state.

  3. TEST-API ATC → SEE (Symbolic Execution Engine)
     First iteration: symbolic variables (X0,X1,...) for all inputs.
     Z3 finds concrete values satisfying all assume() constraints.

  4. Concrete values → Backend execution
     Second iteration with Z3-assigned values calls the actual backend.
     assert() evaluated against real responses. ASSERT FAILED = BUG DETECTED.

  5. CTC (Concrete Test Case) generated showing full sequence with HTTP calls.

DETECTION STRATEGY PER BUG TYPE
================================================================================

  Status code bugs (EB1, EB2, EB3):
    → Add _result == ExpectedCode to spec postcondition (EB1).
    → Factory returns "" on non-201; spec checks _result in dom(map) (EB2, EB3).
    → Either way: wrong status → wrong factory return → assertion fails.

  Arithmetic / calculation bugs (EB4):
    → Add a verification helper endpoint to testRoutes.js (check_order_total).
    → Endpoint recomputes correct value from stored order data and compares.
    → New spec block asserts _result == 1 (pass). Bug gives 400 → caught.

  Missing side-effect bugs (EB5):
    → Spec postcondition checks the state that SHOULD have changed.
    → not_in(buyerEmail, dom(C')) caught directly via get_C() live state fetch.

  Boundary condition bugs (EB6):
    → New factory function fetches current stock and sends exact-stock quantity.
    → Postcondition asserts _result == 200 (acceptance expected).
    → Bug rejects boundary → 400 ≠ 200 → CAUGHT.

  Wrong conditional guard bugs (EB7):
    → Spec postcondition asserts _result in dom(Rev') after createReview.
    → Inverted guard causes HTTP 400 → factory returns "" → in-domain fails.

  Authorization bypass bugs (EB8):
    → Spec block sends request from unauthorized role (buyer deleting product).
    → Postcondition asserts _result == 404 (rejection expected).
    → Bug allows deletion → 200 → 200 ≠ 404 → CAUGHT.
