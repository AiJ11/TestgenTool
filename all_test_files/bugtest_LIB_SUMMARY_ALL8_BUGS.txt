================================================================================
HOW TESTGEN DETECTS BUGS — MECHANISM
================================================================================

  1. SPEC → ATC (Abstract Test Case)
     Formal spec with preconditions and postconditions compiled into a program
     with assume() and assert() statements.

  2. ATC → RewriteGlobalsVisitor → TEST-API ATC
     Global map ops (B[k]=v) rewritten to get/set test-API calls that interact
     with live backend state (get_B, get_S, get_Req, get_Loans).

  3. TEST-API ATC → SEE (Symbolic Execution Engine)
     First iteration: symbolic variables for all inputs.
     Z3 finds concrete values satisfying all assume() constraints.

  4. Concrete values → Backend execution
     Second iteration with Z3-assigned values calls the actual backend.
     assert() evaluated against real responses. ASSERT FAILED = BUG DETECTED.

  5. CTC (Concrete Test Case) generated showing full sequence with HTTP calls.

DETECTION STRATEGY PER BUG TYPE
================================================================================

  Missing side-effect bugs (LB1, LB2, LB4, LB5, LB7, LB8):
    → Spec postconditions check state that SHOULD have changed.
    → get_B() / get_S() / get_Req() / get_Loans() fetch live DB state.
    → Unchanged state contradicts postcondition → assertion fails.
    → LB7/LB8: entity saved with id/code=0 (new Object() pattern),
      which doesn't exist in DB → in-domain check fails at depth=1.

  Wrong conditional guard bugs (LB3):
    → Spec postcondition asserts result is in dom(Loans') after accept.
    → Inverted guard throws exception → HTTP 500 → Num(500) returned.
    → 500 is not a loanId → in-domain check fails.

  Wrong operation bugs (LB6):
    → Spec postcondition asserts entity still EXISTS after update.
    → Delete instead of update removes entity → in-domain check fails.

  Cascade pattern (LB7/LB8):
    → LB8 (addBook → new Book() code=0) cascades into ALL tests needing
      a book: LB1, LB3, LB4, LB6 tests.
    → LB7 (saveStudent → return student, id=0) cascades into ALL tests
      needing a student: LB1, LB3, LB4 tests.
    → This mirrors the EB2 cascade in the ecommerce study.
    → Each bug remains independently detectable and counts for mutation score.
